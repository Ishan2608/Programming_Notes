Installation & Version Pinning

langchain v1langgraph v1.1

LangChain 1.0 (released October 2025) is the first stable release with a commitment to no breaking changes until 2.0. The key new primitive is create_agent — a single, unified function replacing the older create_react_agent, chains, and LCEL pipelines for agent construction.

Package structure in v1

  • langchain — Core agent API: create_agent, @tool, models, messages, middleware
  • langgraph — Agent runtime, graph execution, checkpointers, state management
  • langchain-core — Base abstractions, RunnableConfig, message types
  • langchain-openai / langchain-anthropic / etc. — Provider packages (install only what you use)
  • langchain-classic — Legacy chains, LCEL, LLMChain (moved here for backwards compat)
⚠ Version Pinning — Critical
LangChain's ecosystem moves fast. Always pin exact versions in production. The API in this guide matches langchain==1.2.x, langgraph==1.1.x. If you are reading this in a future date, check docs.langchain.com/oss/python/releases/changelog for breaking changes before upgrading.

Migration note from older LangChain (0.x)

If you have experience with create_react_agent, AgentExecutor, or LCEL chains — those still work via langchain-classic but are not covered here. This guide is v1-native only.

Architecture Overview

Understanding how LangChain v1 fits together saves you a lot of debugging time. create_agent builds a LangGraph graph under the hood — you don't touch the graph directly unless you need deep customisation.

The ReAct loop

An agent runs in a loop: the model decides an action (tool call or final answer), the tool executes, the result returns to the model, and the loop continues until a stop condition. This is the Reason + Act (ReAct) pattern.

Node flow inside create_agent

1. middleware (before_model)Intercept / modify state before LLM call
2. model nodeCall LLM → returns message with tool_calls or final text
3. middleware (wrap_model_call)Wrap model call: dynamic model, token limits
4. tools_conditionRoute: tool calls → tools node, no calls → END
5. tools nodeExecute tool functions in parallel if possible
6. middleware (wrap_tool_call)Wrap tool execution: retry, HITL, PII
7. → back to step 1Loop until no more tool calls

Key concepts

ConceptWhereDescription
AgentStatelanggraphGraph state — holds messages, custom fields. Persisted by checkpointer.
CheckpointerlanggraphPersists state between invocations. Required for multi-turn memory.
Thread IDconfigurableUnique identifier for a conversation. Passed in config at invoke time.
MiddlewarelangchainHooks into the agent loop. Replace prompt templates and chains from v0.
ToolRuntimelangchainInjected into tool functions. Provides access to State, Context, Store.

Models & init_chat_model

init_chat_model is the v1 way to initialise a model. It accepts a model identifier string with automatic provider inference, or an explicit model instance from a provider package. It also exposes a .profile attribute with model capability data.

Model identifier string format

"openai:gpt-4.1"Explicit provider prefix
"gpt-4.1"Auto-inferred as OpenAI
"anthropic:claude-sonnet-4-6"Anthropic (Claude)
"google_genai:gemini-2.0-flash"Google Gemini
"ollama:llama3.3"Local Ollama model
"groq:llama-3.3-70b"Groq (fast inference)

Model profiles (v1.1+)

Models expose a .profile attribute with capability data — used by middleware like SummarizationMiddleware to automatically calculate token fractions. Profile data comes from models.dev.

💡 Model-agnostic design
Build your agent with init_chat_model and an identifier string. Swap models by changing one string — no code changes to tools, middleware, or agent logic. This is the recommended pattern for easy model migration.
code
from langchain.chat_models import init_chat_model
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic

# ── Recommended: identifier string — easy to swap ─────────────────────────────
model = init_chat_model("openai:gpt-4.1")
model = init_chat_model("anthropic:claude-sonnet-4-6")
model = init_chat_model("google_genai:gemini-2.0-flash")

# ── Full control: provider class with explicit params ─────────────────────────
model = ChatOpenAI(
    model="gpt-4.1",
    temperature=0.1,
    max_tokens=4096,
    timeout=60,
    max_retries=2,
)

model = ChatAnthropic(
    model="claude-sonnet-4-6",
    temperature=0,
    max_tokens=8192,
)

# ── Check model profile (v1.1+) ────────────────────────────────────────────────
model = init_chat_model("openai:gpt-4.1")
print(model.profile)
# {
#   "max_input_tokens": 1047576,
#   "structured_output": True,
#   "tool_calling": True,
#   "vision": True,
# }

# ── Custom profile — for models not in models.dev ─────────────────────────────
model = init_chat_model("my-fine-tuned-model", profile={
    "max_input_tokens": 32000,
    "structured_output": False,
    "tool_calling": True,
})

create_agent — Core API

v1 — replaces create_react_agent

create_agent is the single entry point for building agents in LangChain v1. It returns a compiled LangGraph graph. You invoke it the same way you'd invoke any graph: .invoke(), .ainvoke(), .stream(), or .astream().

create_agent() parameters

ParameterTypeDescription
modelstr | BaseChatModelModel identifier string or model instance. Required.
toolslist[tool]Tools available to the agent. Empty list = LLM-only node (no tool loop).
system_promptstr | SystemMessageSystem message prepended to every conversation. Supports SystemMessage instances (v1.1+) for cache control.
namestrAgent name — used in multi-agent setups for routing and logging.
checkpointerBaseCheckpointSaverEnables short-term memory persistence across invocations. Required for multi-turn chat.
state_schemaAgentState subclassCustom state with additional fields beyond the default messages key.
middlewarelist[Middleware]Ordered list of middleware applied to the agent loop.
response_formattype | ToolStrategy | ProviderStrategySchema for structured output. Auto-selects best strategy based on model profile.
context_schemadataclass typeType for immutable runtime context passed in config["configurable"]["context"].
storeBaseStoreLong-term memory store accessible to tools and middleware.

Invocation

Input is always a dict with a "messages" key. Messages can be dicts (shorthand) or proper message objects. Output is the full final state dict.

code
from langchain.agents import create_agent
from langchain.tools import tool
import asyncio

# ── Minimal agent — just a model ──────────────────────────────────────────────
agent = create_agent("openai:gpt-4.1")
result = agent.invoke({"messages": [{"role": "user", "content": "Hello"}]})
print(result["messages"][-1].content)

# ── Agent with tools and system prompt ────────────────────────────────────────
@tool
def get_weather(location: str) -> str:
    """Get current weather for a location."""
    return f"Weather in {location}: 28°C, sunny"

agent = create_agent(
    "anthropic:claude-sonnet-4-6",
    tools=[get_weather],
    system_prompt="You are a helpful weather assistant. Always give concise answers.",
)

# ── Sync invoke ───────────────────────────────────────────────────────────────
result = agent.invoke({
    "messages": [{"role": "user", "content": "What's the weather in Delhi?"}]
})
print(result["messages"][-1].content)

# ── Async invoke (preferred for production) ───────────────────────────────────
async def run():
    result = await agent.ainvoke({
        "messages": [{"role": "user", "content": "Weather in Mumbai?"}]
    })
    return result["messages"][-1].content

asyncio.run(run())
▶ output
The weather in Delhi is currently 28°C and sunny.

Tools — @tool Decorator

Tools are the actions your agent can take. The @tool decorator converts a Python function into a LangChain tool. The docstring becomes the tool description the model sees to decide when to use it — make it precise and actionable.

Rules for good tools

  • Type hints are required — they define the input schema sent to the model
  • Use snake_case names — some providers reject names with spaces or special characters
  • Reserved names: Do not use runtime or config as argument names — they are injected by the framework
  • Both sync and async functions work — async def for I/O-bound tools

Tool return types

Return typeEffect
strReturned as-is as the tool message content
dict / PydanticSerialised to JSON string in tool message
CommandUpdate agent state AND optionally resume from an interrupt
⚠ Docstring quality matters
The model decides which tool to call based on the description. A vague docstring like "Search for something" leads to wrong tool selection. Be specific: what it does, when to use it, and what format the input should be.
code
from langchain.tools import tool
from pydantic import BaseModel, Field
from langgraph.types import Command
from typing import Literal

# ── Basic tool — docstring = description the model sees ───────────────────────
@tool
def search_web(query: str) -> str:
    """Search the web for current information. Use for recent events, facts, or
    anything that requires up-to-date data. Input should be a search query string."""
    return f"Search results for: {query}"

# ── Custom name and description ───────────────────────────────────────────────
@tool("calculator", description="Evaluate a Python math expression. Input: a string like '2 ** 10 + 5'.")
def calc(expression: str) -> str:
    """Evaluate arithmetic."""
    return str(eval(expression, {"__builtins__": {}}, {}))

# ── Pydantic schema for complex inputs ────────────────────────────────────────
class DatabaseQuery(BaseModel):
    table:      str   = Field(description="Table name to query")
    filter_col: str   = Field(description="Column name to filter on")
    filter_val: str   = Field(description="Value to filter for")
    limit:      int   = Field(10, ge=1, le=100, description="Max rows to return")

@tool(args_schema=DatabaseQuery)
def query_database(table: str, filter_col: str, filter_val: str, limit: int = 10) -> str:
    """Query a database table with a column filter. Returns matching rows as JSON."""
    return f"[{limit} rows from {table} where {filter_col}={filter_val}]"

# ── Async tool — for I/O-bound work (HTTP calls, DB queries) ──────────────────
import httpx

@tool
async def fetch_url(url: str) -> str:
    """Fetch the content of a URL. Returns the response body text. Use for
    reading web pages, APIs, or any HTTP endpoint."""
    async with httpx.AsyncClient() as client:
        response = await client.get(url, timeout=10)
        return response.text[:2000]   # truncate for context

# ── Command return — update agent state ───────────────────────────────────────
@tool
def set_language_preference(language: str) -> Command:
    """Set the user's preferred response language. Call when user explicitly
    requests responses in a different language."""
    return Command(update={"preferred_language": language})

ToolRuntime — State, Context & Store

Add runtime: ToolRuntime to any tool signature to access runtime information. The parameter is automatically injected by the framework and is hidden from the model — it won't appear in the tool's schema.

ToolRuntime components

AttributeTypeDescription
runtime.statedictRead the current agent state. Includes "messages" and any custom fields.
runtime.contextdataclass instanceImmutable runtime context passed at invocation. User ID, session info, permissions.
runtime.storeBaseStoreLong-term memory store. Survives across conversations.
runtime.stream_writercallableEmit real-time progress updates during a long-running tool.

Context pattern

Define a @dataclass as your context schema, pass it to create_agent(context_schema=...), and inject at invocation time via config["configurable"]["context"]. Use for per-request user identity, API keys, or tenant info.

💡 State vs Context vs Store
State = this conversation's scratch pad (resets per thread). Context = request-scoped config (who is calling). Store = long-term DB (persists forever).
code
from dataclasses import dataclass
from langchain.tools import tool, ToolRuntime
from langchain.agents import create_agent
from langchain.messages import HumanMessage
from langgraph.types import Command

# ── Define context schema ─────────────────────────────────────────────────────
@dataclass
class UserContext:
    user_id:      str
    account_tier: str   = "free"
    tenant_id:    str   = "default"

# ── Read from state ───────────────────────────────────────────────────────────
@tool
def get_conversation_summary(runtime: ToolRuntime) -> str:
    """Summarise the key points discussed so far in this conversation."""
    messages = runtime.state["messages"]
    human_msgs = [m.content for m in messages if isinstance(m, HumanMessage)]
    return f"User has asked {len(human_msgs)} questions. Last: '{human_msgs[-1] if human_msgs else 'none'}'"

# ── Read context (immutable — who is calling) ─────────────────────────────────
@tool
def get_account_balance(runtime: ToolRuntime[UserContext]) -> str:
    """Get the current user's account balance."""
    user_id = runtime.context.user_id
    tier    = runtime.context.account_tier
    # look up user_id in your DB
    return f"User {user_id} ({tier}): ₹12,500 balance"

# ── Update state via Command ───────────────────────────────────────────────────
@tool
def mark_task_done(task_id: str, runtime: ToolRuntime) -> Command:
    """Mark a task as completed in the current session."""
    completed = runtime.state.get("completed_tasks", [])
    return Command(update={"completed_tasks": completed + [task_id]})

# ── Stream progress from a long-running tool ──────────────────────────────────
@tool
async def process_large_file(file_path: str, runtime: ToolRuntime) -> str:
    """Process a large file. Emits progress updates during processing."""
    for i in range(1, 6):
        runtime.stream_writer({"progress": f"Processing chunk {i}/5…"})
        await asyncio.sleep(1)
    return "File processing complete"

# ── Create agent with context schema ──────────────────────────────────────────
agent = create_agent(
    "openai:gpt-4.1",
    tools=[get_account_balance, mark_task_done],
    context_schema=UserContext,
)

# ── Invoke — inject context at call time ──────────────────────────────────────
result = agent.invoke(
    {"messages": [{"role": "user", "content": "What's my balance?"}]},
    config={"configurable": {"context": UserContext(user_id="u_123", account_tier="premium")}}
)

Short-term Memory & Checkpointers

By default, each agent.invoke() call is stateless — no memory of previous turns. Add a checkpointer to persist conversation state across calls. A thread_id in the config identifies which conversation thread to resume.

Checkpointer options

InMemorySaverDev/testing only — lost on process restart
SqliteSaverSingle-process prod / serverless
PostgresSaverMulti-worker production (recommended)
AsyncPostgresSaverAsync-native Postgres for async agents

Custom state

Extend AgentState to add fields beyond messages. The custom state is persisted alongside messages and accessible in tools via runtime.state.

⚠ Thread ID is your conversation ID
All calls with the same thread_id share memory. Use a new unique thread_id per user session. Never reuse thread IDs across different users.
code
from langchain.agents import create_agent, AgentState
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.checkpoint.postgres import PostgresSaver
from langchain.tools import tool

@tool
def get_user_info(user_id: str) -> str:
    """Fetch user profile information from the system."""
    return f"User {user_id}: Alice, Premium subscriber, Delhi"

# ── Dev: in-memory checkpointer ───────────────────────────────────────────────
agent = create_agent(
    "openai:gpt-4.1",
    tools=[get_user_info],
    checkpointer=InMemorySaver(),
)

config = {"configurable": {"thread_id": "user-alice-session-1"}}

agent.invoke({"messages": [{"role": "user", "content": "Hi, I'm Alice"}]}, config)
result = agent.invoke({"messages": [{"role": "user", "content": "What's my name?"}]}, config)
print(result["messages"][-1].content)  # "Your name is Alice"

# ── Production: PostgreSQL checkpointer ───────────────────────────────────────
DB_URI = "postgresql://user:password@localhost:5432/mydb"

with PostgresSaver.from_conn_string(DB_URI) as checkpointer:
    checkpointer.setup()   # creates tables on first run
    agent = create_agent("openai:gpt-4.1", tools=[get_user_info], checkpointer=checkpointer)

# ── Custom state schema ────────────────────────────────────────────────────────
class MyAgentState(AgentState):
    user_name:          str        = ""
    completed_tasks:    list[str] = []
    preferred_language: str        = "english"

agent = create_agent(
    "openai:gpt-4.1",
    tools=[get_user_info],
    checkpointer=InMemorySaver(),
    state_schema=MyAgentState,
)

# Pass custom state fields at invocation
result = agent.invoke(
    {"messages": [{"role": "user", "content": "Hello"}],
     "user_name": "Alice", "preferred_language": "hindi"},
    config
)

Structured Output

Use response_format in create_agent to get validated, typed data back from the agent instead of a raw text message. The structured result is in result["structured_response"].

Strategy selection

Pass a schema type directly and LangChain auto-selects the strategy based on the model's .profile:

  • ProviderStrategy — used when model supports native structured output (OpenAI, Anthropic, Gemini). Enforced at the API level.
  • ToolStrategy — uses a special tool call to extract the schema. Works with any model that supports tool calling.

Supported schema types

Pydantic BaseModelReturns validated model instance
Python dataclassReturns dict
TypedDictReturns dict
Union typesModel picks the best-matching schema
code
from pydantic import BaseModel, Field
from typing import Literal
from langchain.agents import create_agent
from langchain.agents.structured_output import ProviderStrategy, ToolStrategy

# ── Define output schema ──────────────────────────────────────────────────────
class TicketClassification(BaseModel):
    """Classification of a support ticket."""
    category:  Literal["billing", "technical", "shipping", "other"]
    priority:  Literal["low", "medium", "high", "urgent"]
    sentiment: Literal["positive", "neutral", "negative"]
    summary:   str = Field(description="One-sentence summary of the ticket")
    tags:      list[str] = Field(description="Relevant tags for routing")

# ── Auto-select strategy (recommended) ────────────────────────────────────────
agent = create_agent(
    "openai:gpt-4.1",
    system_prompt="You are a support ticket classifier.",
    response_format=TicketClassification,   # auto-selects ProviderStrategy for OpenAI
)

result = agent.invoke({"messages": [{"role": "user",
    "content": "My order hasn't arrived after 3 weeks! This is unacceptable."}]})

ticket = result["structured_response"]
print(type(ticket))       # <class 'TicketClassification'>
print(ticket.category)   # shipping
print(ticket.priority)   # urgent
print(ticket.sentiment)  # negative

# ── Explicit ProviderStrategy with strict mode (v1.2+) ────────────────────────
agent = create_agent(
    "openai:gpt-4.1",
    response_format=ProviderStrategy(schema=TicketClassification, strict=True),
)

# ── ToolStrategy — for models without native structured output ─────────────────
agent = create_agent(
    "ollama:llama3.3",
    response_format=ToolStrategy(
        schema=TicketClassification,
        handle_errors=True,   # retry on validation failure
    ),
)

# ── Union type — model picks the right schema ─────────────────────────────────
class Approved(BaseModel):
    decision: Literal["approved"]
    reason: str

class Rejected(BaseModel):
    decision: Literal["rejected"]
    reason: str
    appeal_deadline: str

agent = create_agent("openai:gpt-4.1", response_format=Approved | Rejected)
▶ output
TicketClassification(
  category='shipping', priority='urgent', sentiment='negative',
  summary='Customer reports order not delivered after 3 weeks.',
  tags=['late-delivery', 'order-tracking', 'escalation']
)

Streaming

Use .stream() or .astream() to receive events as the agent runs — token by token from the model, tool call events, and state updates. LangGraph v1.1 introduces version="v2" for typed StreamPart output.

Stream modes

modeWhat you get
"values"Full state snapshot after each node runs
"updates"Only the state delta from each node (lighter)
"messages"Individual message chunks (token streaming from model)
"custom"Custom events emitted via runtime.stream_writer() inside tools
code
from langchain.agents import create_agent
from langchain.tools import tool

agent = create_agent("openai:gpt-4.1", tools=[])

input_msg = {"messages": [{"role": "user", "content": "Write a haiku about Python."}]}

# ── Token streaming — "messages" mode ─────────────────────────────────────────
for chunk in agent.stream(input_msg, stream_mode="messages"):
    if chunk[1]["langgraph_node"] == "model":
        print(chunk[0].content, end="", flush=True)

# ── State updates mode — see each node's output ───────────────────────────────
for step in agent.stream(input_msg, stream_mode="updates"):
    for node, state_delta in step.items():
        print(f"Node '{node}' updated state keys: {list(state_delta.keys())}")

# ── Async streaming (preferred in production) ─────────────────────────────────
async def stream_agent():
    async for chunk in agent.astream(input_msg, stream_mode="messages"):
        msg, meta = chunk
        if meta["langgraph_node"] == "model" and msg.content:
            print(msg.content, end="", flush=True)

# ── LangGraph v1.1: typed StreamPart with version="v2" ───────────────────────
for part in agent.stream(input_msg, version="v2", stream_mode="messages"):
    print(part.type)   # "messages" | "updates" | "values" | "custom"
    print(part.ns)     # node namespace
    print(part.data)   # typed chunk data

# ── Custom events from tools via stream_writer ────────────────────────────────
@tool
async def long_analysis(topic: str, runtime) -> str:
    """Analyse a topic in depth."""
    runtime.stream_writer({"status": "Gathering data…"})
    await asyncio.sleep(1)
    runtime.stream_writer({"status": "Running analysis…"})
    await asyncio.sleep(1)
    return f"Analysis of {topic} complete"

agent2 = create_agent("openai:gpt-4.1", tools=[long_analysis])

async for chunk in agent2.astream(
    {"messages": [{"role": "user", "content": "Analyse quantum computing trends"}]},
    stream_mode=["messages", "custom"]
):
    print(chunk)

Middleware — Custom Hooks

Middleware is the v1 replacement for prompt templates, chains, and custom agent executors. Each middleware is a decorator-annotated function that intercepts a specific point in the agent loop. They compose like onion layers.

Middleware hook types

DecoratorWhen it runsUse for
@before_modelBefore LLM is calledInject context into messages, trim history, add system info
@after_modelAfter LLM respondsLog responses, validate model output, post-process messages
@wrap_model_callWraps the entire model callDynamic model selection, token budget enforcement, retries
@wrap_tool_callWraps each tool executionTool-level logging, PII masking, approval gates, retries
💡 Middleware vs old prompt templates
In LangChain 0.x you'd use a PromptTemplate to inject date/time into the system prompt. In v1, do it in a @before_model middleware that modifies the messages list directly.
code
from datetime import datetime
from langchain.agents import create_agent, AgentState
from langchain.agents.middleware import (
    before_model, after_model, wrap_model_call, wrap_tool_call,
    ModelRequest, ModelResponse
)
from langchain.messages import SystemMessage
from langgraph.runtime import Runtime
from typing import Any, Callable

# ── @before_model — inject dynamic context into messages ──────────────────────
@before_model
def inject_current_time(state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
    """Prepend a system message with the current date/time."""
    now = datetime.now().strftime("%A, %B %d %Y %H:%M")
    time_msg = SystemMessage(content=f"Current date/time: {now}. Use this for time-sensitive queries.")
    return {"messages": [time_msg] + state["messages"]}

# ── @after_model — log or inspect model output ────────────────────────────────
@after_model
def log_model_calls(state: AgentState, runtime: Runtime) -> None:
    last_msg = state["messages"][-1]
    tool_call_count = len(getattr(last_msg, "tool_calls", []))
    print(f"Model responded. Tool calls: {tool_call_count}")

# ── @wrap_model_call — dynamic model selection ────────────────────────────────
from langchain_openai import ChatOpenAI

mini_model  = ChatOpenAI(model="gpt-4.1-mini")
full_model  = ChatOpenAI(model="gpt-4.1")

@wrap_model_call
def smart_model_router(
    request: ModelRequest,
    handler: Callable[[ModelRequest], ModelResponse]
) -> ModelResponse:
    """Use mini for simple questions, full for complex reasoning."""
    last_user_msg = request.state["messages"][-1].content
    is_complex = len(last_user_msg.split()) > 30 or "analyse" in last_user_msg.lower()
    chosen = full_model if is_complex else mini_model
    return handler(request.override(model=chosen))

# ── @wrap_tool_call — approval gate, audit log ────────────────────────────────
@wrap_tool_call
def audit_tool_calls(tool_call, handler):
    print(f"[AUDIT] Tool called: {tool_call.name} | Args: {tool_call.args}")
    result = handler(tool_call)
    print(f"[AUDIT] Tool result: {str(result)[:100]}")
    return result

# ── Compose: middleware applied in order ──────────────────────────────────────
agent = create_agent(

    middleware=[inject_current_time, log_model_calls, smart_model_router, audit_tool_calls],

Prebuilt Middleware

LangChain ships production-ready middleware for the most common agent concerns. Import and drop them in — no boilerplate required.

Available prebuilt middleware

ClassDescription
SummarizationMiddlewareSummarise old messages when approaching token limit. Prevents context overflow.
HumanInTheLoopMiddlewarePause before tool execution for human approval.
ModelCallLimitMiddlewareStop after N model calls. Prevents runaway agents.
ToolCallLimitMiddlewareLimit total tool invocations per run.
ModelFallbackMiddlewareTry alternate models if primary fails.
PIIDetectionMiddlewareDetect and mask/block PII in inputs and outputs.
ToolRetryMiddlewareRetry failed tool calls with exponential backoff.
ModelRetryMiddlewareRetry failed model calls with exponential backoff. (v1.1+)
ContentModerationMiddlewareOpenAI moderation check on inputs/outputs. (v1.1+)

from langchain.agents import create_agent
from langchain.agents.middleware import (
    SummarizationMiddleware,
    ModelCallLimitMiddleware,
    ToolCallLimitMiddleware,
    ModelFallbackMiddleware,
    PIIDetectionMiddleware,
    ToolRetryMiddleware,
    ModelRetryMiddleware,        # v1.1+
    ContentModerationMiddleware,  # v1.1+
)

# ── Summarization — auto-compress old messages at token threshold ──────────────
summarization = SummarizationMiddleware(
    model="openai:gpt-4.1-mini",       # cheap model for summaries
    trigger="tokens", 4000,            # summarise when > 4000 tokens
    keep=("messages", 20),             # keep last 20 messages verbatim
)

model_limit = ModelCallLimitMiddleware(limit=15)   # max 15 LLM calls

)

pii = PIIDetectionMiddleware(
    entities=["EMAIL_ADDRESS", "PHONE_NUMBER", "CREDIT_CARD"],
    action="mask",    # "mask" | "block" | "flag"

tool_retry  = ToolRetryMiddleware(max_retries=3, backoff_factor=2.0)

    tools=[],

Human-in-the-Loop

Pause agent execution before a tool call, let a human approve/reject/modify it, then resume. This requires a checkpointer — the agent state is saved at the interrupt point and restored on resume.

Two interrupt patterns

  • HumanInTheLoopMiddleware — intercepts all (or filtered) tool calls, emits an interrupt, waits for human input via a Command(resume=...)
  • Manual interrupt via interrupt() — call from inside a tool or middleware for custom approval flows
⚠ Checkpointer is required
Human-in-the-loop does not work without a checkpointer. The agent state must be saved before the interrupt so it can be resumed. Use InMemorySaver for dev, PostgresSaver for production.
code
from langchain.agents import create_agent
from langchain.agents.middleware import HumanInTheLoopMiddleware
from langchain.tools import tool
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import Command

@tool
def delete_record(record_id: str) -> str:
    """Permanently delete a record from the database. This action cannot be undone."""
    return f"Record {record_id} deleted"

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """Send an email to a recipient."""
    return f"Email sent to {to}"

checkpointer = InMemorySaver()

agent = create_agent(
    "openai:gpt-4.1",
    tools=[delete_record, send_email],
    checkpointer=checkpointer,
    middleware=[
        HumanInTheLoopMiddleware(
            tools=["delete_record", "send_email"],  # only these need approval
        )
    ]
)

config = {"configurable": {"thread_id": "hitl-demo"}}

# ── Step 1: agent runs, hits delete_record, emits interrupt ───────────────────
result = agent.invoke(
    {"messages": [{"role": "user", "content": "Delete record ID 42"}]},
    config,
)

# result["__interrupt__"] is set — agent is paused
interrupt_info = result.get("__interrupt__")
print(f"⏸  Pending approval: {interrupt_info}")

# ── Step 2: human reviews and approves ────────────────────────────────────────
human_decision = "approve"   # or "reject" or a modified args dict

# Resume the agent with the human's decision
final_result = agent.invoke(
    Command(resume=human_decision),
    config,
)
print(final_result["messages"][-1].content)

# ── Manual interrupt inside a tool ────────────────────────────────────────────
from langgraph.types import interrupt

@tool
def transfer_funds(amount: float, to_account: str) -> str:
    """Transfer funds to another account. Requires human approval for amounts over ₹10,000."""
    if amount > 10000:
        approval = interrupt({
            "message": f"Approve transfer of ₹{amount} to {to_account}?",
            "amount": amount,
            "to_account": to_account
        })
        if approval != "approve":
            return "Transfer rejected by human"
    return f"Transferred ₹{amount} to {to_account}"

Long-term Memory (Store)

The Store provides cross-session, cross-thread persistent memory. Unlike a checkpointer (which saves state per-thread), the store is a shared key-value database accessible across all agent invocations. Tools read and write it via runtime.store.

Store key structure

Keys are namespaced tuples: (namespace, key). Use the namespace to scope by user, tenant, or domain. Example: (f"user:{user_id}:preferences", "theme").

Store options

InMemoryStoreDev/testing — not persistent
PostgresStoreProduction — persistent, queryable
AsyncPostgresStoreAsync-native Postgres store
code
from langchain.tools import tool, ToolRuntime
from langchain.agents import create_agent
from langgraph.store.memory import InMemoryStore

store = InMemoryStore()

# ── Tool that reads from long-term memory ─────────────────────────────────────
@tool
def get_user_preferences(runtime: ToolRuntime) -> str:
    """Retrieve the current user's stored preferences."""
    user_id = runtime.context.user_id
    namespace = (f"user:{user_id}", "preferences")
    item = runtime.store.get(namespace, "settings")
    if item:
        return f"Preferences: {item.value}"
    return "No preferences saved yet."

# ── Tool that writes to long-term memory ──────────────────────────────────────
@tool
def save_user_preference(key: str, value: str, runtime: ToolRuntime) -> str:
    """Save a user preference. Persists across sessions.
    Args:
        key: preference name (e.g. 'language', 'currency', 'timezone')
        value: preference value
    """
    user_id   = runtime.context.user_id
    namespace = (f"user:{user_id}", "preferences")

    existing  = runtime.store.get(namespace, "settings")
    settings  = existing.value if existing else {}
    settings[key] = value
    runtime.store.put(namespace, "settings", settings)
    return f"Saved: {key} = {value}"

# ── Semantic search in store (PostgresStore with embedding support) ────────────
@tool
async def search_memories(query: str, runtime: ToolRuntime) -> str:
    """Search past conversations and saved memories by semantic similarity."""
    user_id = runtime.context.user_id
    results = await runtime.store.asearch(
        (f"user:{user_id}", "memories"),
        query=query,
        limit=5,
    )
    return "\n".join([item.value["text"] for item in results])

# ── Wire store into agent ─────────────────────────────────────────────────────
from dataclasses import dataclass

@dataclass
class Ctx:
    user_id: str

agent = create_agent(
    "openai:gpt-4.1",
    tools=[get_user_preferences, save_user_preference],
    store=store,
    context_schema=Ctx,
)

agent.invoke(
    {"messages": [{"role": "user", "content": "Save my currency preference as INR"}]},
    config={"configurable": {"context": Ctx(user_id="u_alice")}}
)

Multi-Agent Systems

Large tasks are best split across specialised agents. A supervisor agent routes tasks; subagents handle domain-specific work. In v1, agents can call other agents as tools using the SubagentMiddleware or by converting an agent to a tool.

Patterns

  • Supervisor + workers — one orchestrating agent, specialised workers as tools
  • Peer-to-peer — agents call each other as tools when needed
  • Sequential pipeline — output of one agent feeds as input to the next
💡 Name your agents
Set name= on each create_agent call. The name is used in LangSmith traces, the subagent middleware routing, and logging. Makes multi-agent traces readable.
code
from langchain.agents import create_agent
from langchain.tools import tool
from langchain_openai import ChatOpenAI

# ── Specialised subagents ─────────────────────────────────────────────────────
@tool
def search_web(q: str) -> str:
    """Search the web."""
    return f"Results for: {q}"

@tool
def run_python(code: str) -> str:
    """Execute Python code and return output."""
    import io, contextlib
    buf = io.StringIO()
    with contextlib.redirect_stdout(buf):
        exec(code, {})
    return buf.getvalue()

research_agent = create_agent(
    "openai:gpt-4.1",
    tools=[search_web],
    system_prompt="You are a research specialist. Find accurate, cited information.",
    name="research_agent",
)

analysis_agent = create_agent(
    "openai:gpt-4.1",
    tools=[run_python],
    system_prompt="You are a data analyst. Analyse data and produce insights.",
    name="analysis_agent",
)

# ── Convert subagents to tools for the supervisor ─────────────────────────────
@tool
async def delegate_research(question: str) -> str:
    """Delegate a research question to the research specialist agent.
    Use when the task requires finding information, facts, or current events."""
    result = await research_agent.ainvoke({
        "messages": [{"role": "user", "content": question}]
    })
    return result["messages"][-1].content

@tool
async def delegate_analysis(task: str, data: str) -> str:
    """Delegate a data analysis task to the analysis specialist agent.
    Use when you have data that needs statistical analysis or visualisation."""
    result = await analysis_agent.ainvoke({
        "messages": [{"role": "user", "content": f"{task}\n\nData:\n{data}"}]
    })
    return result["messages"][-1].content

# ── Supervisor — routes tasks to subagents ────────────────────────────────────
supervisor = create_agent(
    "openai:gpt-4.1",
    tools=[delegate_research, delegate_analysis],
    system_prompt="You are a project supervisor. Break complex tasks into sub-tasks "
                  "and delegate them to the appropriate specialist agents.",
    name="supervisor",
)

result = await supervisor.ainvoke({
    "messages": [{"role": "user",
        "content": "Research India's GDP growth in 2024 and then analyse the trend"}]
})

Model Context Protocol (MCP)

MCP is an open standard for connecting agents to external tool servers. Use langchain-mcp-adapters to load tools from any MCP server — local (stdio subprocess) or remote (HTTP). The tools integrate directly with create_agent.

Transport types

stdioLocal subprocess — spawns server as child process
httpRemote HTTP server (streamable-http)
sseServer-Sent Events (deprecated in MCP spec — prefer http)
⚠ Stateless by default
MultiServerMCPClient creates a fresh session per tool call. Use client.session() context manager for stateful MCP servers that maintain state across calls.
code
import asyncio
from langchain_mcp_adapters.client import MultiServerMCPClient
from langchain.agents import create_agent

# ── Connect to MCP servers ────────────────────────────────────────────────────
client = MultiServerMCPClient({
    "filesystem": {
        "transport": "stdio",
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
    },
    "web_search": {
        "transport": "http",
        "url": "http://localhost:8080/mcp",
        "headers": {"Authorization": "Bearer MY_API_KEY"},
    },
})

async def main():
    # Load all tools from all connected servers
    tools = await client.get_tools()
    print([t.name for t in tools])

    agent = create_agent("anthropic:claude-sonnet-4-6", tools)

    result = await agent.ainvoke({
        "messages": [{"role": "user", "content": "List the files in /tmp and search the web for Python 3.14 news"}]
    })
    print(result["messages"][-1].content)

asyncio.run(main())

Retrieval-Augmented Generation (RAG)

RAG gives agents access to a document corpus via a retriever tool. The agent decides when to call the retriever based on the query — no need for a hardcoded retrieval step. This is the v1 pattern: retrieval as just another tool.

Vector store options

ChromaLocal, easy setup — pip install chromadb
PineconeManaged cloud vector DB
pgvectorPostgres extension — good if already using Postgres
FAISSIn-memory, fast — pip install faiss-cpu
code
from langchain.tools import tool
from langchain.agents import create_agent
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma
from langchain_community.document_loaders import PyPDFLoader, WebBaseLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter

# ── Build vector store from documents ─────────────────────────────────────────
loader     = PyPDFLoader("company_docs.pdf")
documents  = loader.load()

splitter   = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
chunks     = splitter.split_documents(documents)

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(
    chunks, embeddings,
    persist_directory="./chroma_db"   # persists to disk
)
retriever  = vectorstore.as_retriever(search_kwargs={"k": 5})

# ── Wrap retriever as a tool ───────────────────────────────────────────────────
@tool
def search_company_docs(query: str) -> str:
    """Search the company knowledge base for information about policies,
    products, procedures, and guidelines. Use this for any question about
    internal company information."""
    docs = retriever.invoke(query)
    if not docs:
        return "No relevant documents found."
    return "\n\n---\n\n".join([
        f"[{d.metadata.get('source', 'doc')}, page {d.metadata.get('page', '?')}]\n{d.page_content}"
        for d in docs
    ])

# ── Also load from URLs ────────────────────────────────────────────────────────
web_loader = WebBaseLoader(["https://company.com/docs", "https://company.com/faq"])
web_docs   = web_loader.load()
web_chunks = splitter.split_documents(web_docs)
vectorstore.add_documents(web_chunks)   # add to same store

# ── Create RAG agent ──────────────────────────────────────────────────────────
rag_agent = create_agent(
    "anthropic:claude-sonnet-4-6",
    tools=[search_company_docs],
    system_prompt="You are a helpful company assistant. Use the search_company_docs "
                  "tool to find accurate information before answering. "
                  "Always cite the source document in your response.",
)

result = rag_agent.invoke({
    "messages": [{"role": "user", "content": "What is our refund policy?"}]
})

Testing Agents

Testing agents is harder than testing APIs because outputs are non-deterministic. The strategy: test the structure (did the right tool get called? did the schema validate?), not the exact text. Mock expensive tools and use deterministic models where possible.

Testing checklist

  • Unit test tools in isolation — just call the Python function directly
  • Mock tool return values — control what the agent "sees" from tools
  • Test tool selection — assert which tool was called, not the LLM text
  • Test structured output — validate schema fields, not prose
  • Set temperature=0 — makes LLM responses more deterministic in tests
code
# tests/test_agent.py ──────────────────────────────────────────────────────────
import pytest
from unittest.mock import patch, MagicMock
from pydantic import BaseModel
from langchain.agents import create_agent
from langchain.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.checkpoint.memory import InMemorySaver

# ── Unit test a tool in isolation ──────────────────────────────────────────────
@tool
def get_weather(city: str) -> str:
    """Get weather for a city."""
    return f"{city}: 25°C"

def test_tool_directly():
    """Test tool function logic without involving LLM."""
    result = get_weather.invoke({"city": "Delhi"})
    assert "Delhi" in result
    assert "°C" in result

# ── Test with deterministic temperature=0 model ───────────────────────────────
def test_agent_uses_weather_tool():
    model = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
    agent = create_agent(model, tools=[get_weather])
    result = agent.invoke({"messages": [{"role": "user", "content": "Weather in Mumbai?"}]})

    # Check that a tool was called
    tool_calls = [m for m in result["messages"] if hasattr(m, "tool_calls") and m.tool_calls]
    assert len(tool_calls) > 0
    assert tool_calls[0].tool_calls[0]["name"] == "get_weather"

# ── Test with mocked tool (no real API calls) ──────────────────────────────────
def test_agent_with_mock_tool():
    with patch.object(get_weather, "func", return_value="Mocked: 30°C, rainy"):
        model = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
        agent = create_agent(model, tools=[get_weather])
        result = agent.invoke({"messages": [{"role": "user", "content": "Weather in Delhi?"}]})
        final = result["messages"][-1].content
        assert "30" in final or "rainy" in final

# ── Test structured output schema ─────────────────────────────────────────────
class SentimentResult(BaseModel):
    sentiment: str
    score: float

def test_structured_output():
    model = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
    agent = create_agent(model, response_format=SentimentResult)
    result = agent.invoke({"messages": [{"role": "user", "content": "I love this product!"}]})
    output = result["structured_response"]
    assert isinstance(output, SentimentResult)
    assert output.sentiment in ["positive", "neutral", "negative"]
    assert 0.0 <= output.score <= 1.0

# ── Test multi-turn memory ─────────────────────────────────────────────────────
def test_memory_persists():
    checkpointer = InMemorySaver()
    model = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
    agent = create_agent(model, tools=[], checkpointer=checkpointer)
    cfg   = {"configurable": {"thread_id": "test-thread-1"}}

    agent.invoke({"messages": [{"role": "user", "content": "My name is Alice"}]}, cfg)
    result = agent.invoke({"messages": [{"role": "user", "content": "What is my name?"}]}, cfg)
    assert "Alice" in result["messages"][-1].content

Observability with LangSmith

LangSmith is LangChain's tracing and monitoring platform. Enable it with two environment variables — zero code changes required. Every agent run is automatically traced: model calls, tool invocations, latency, token usage, and errors.

What LangSmith tracks

Full trace treeEvery node run — model, tools, middleware
Token usageInput / output tokens per model call
LatencyTime spent in each node
ErrorsExceptions with full stack traces
FeedbackHuman thumbs up/down on runs
EvalsAutomated LLM-as-judge scoring

Pass tags and metadata in the run config to annotate traces for filtering in LangSmith. Essential for production monitoring and A/B testing different prompts or models.
