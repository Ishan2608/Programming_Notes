Introduction to Big Data

01 Evolution of Big Data

1800–1950

Mechanical & Early Digital Processing

  • Joseph Marie Jacquard (1801) introduced punched cards to control textile looms — each hole directed the loom's needle, one of the earliest forms of programmatic data input.
  • Herman Hollerith (1890) adapted punched cards for the U.S. Census, reducing a 10-year tabulation effort to just 3 months.
  • Alan Turing (1936) proposed the Turing Machine — a theoretical model that became the foundation of all modern computing.
  • ENIAC (1946), the first general-purpose electronic computer, processed simple numeric data at speeds impossible for humans.
1950–1990

Rise of Digital Data & Databases

  • IBM 305 RAMAC (1956) — the first hard disk drive, storing 5MB of data despite weighing over a ton.
  • Edgar F. Codd (1970) proposed the relational model — users should see data as tables, not care about pointers or trees — giving birth to SQL.
  • Oracle (1977) and IBM DB2 (1983) became the commercial cornerstones of structured data management.
1980s–1990s

Personal Computers & the Internet

  • Spreadsheets — VisiCalc (1979), Lotus 1-2-3 (1983), Microsoft Excel (1985) — brought data analysis to non-programmers.
  • Tim Berners-Lee's World Wide Web (1991) enabled global data sharing, generating the first waves of unstructured internet data.
2000s

The Data Explosion

  • Social media — Facebook, Twitter, YouTube — created massive volumes of unstructured data that relational databases couldn't scale to handle.
  • Google developed GFS and MapReduce, inspiring Yahoo to create Apache Hadoop.
  • Cloud platforms — AWS, Google Cloud, Microsoft Azure — enabled businesses to store and process data at scale without owning physical infrastructure.
2010s

AI, Machine Learning & Real-Time Analytics

  • Apache Kafka and Apache Spark enabled real-time streaming analytics at massive scale.
  • ML models trained on Big Data enabled powerful predictions — Netflix's recommendation engine uses viewing history across millions of users.
  • Blockchain emerged as a mechanism for secure, tamper-proof distributed data transactions.
2020s

5G, IoT & Quantum Computing

  • A 4G tower handles ~2,000 IoT devices per sq. km. A 5G tower handles up to 1 million devices.
  • Computing power hierarchy evolved: CPU → GPU → QPU (Quantum Processing Units).
  • QPUs process all bits simultaneously — solving complex optimization problems that would take classical computers millions of years.

02 Types of Data

Based on Structure

Structured Data
Organized in rows and columns with a predefined schema. Stored in relational databases (SQL, Excel). Example: employee table with ID, Name, Department, Salary columns.
Semi-Structured
No rigid table format but contains tags/keys for organizational hierarchy. Common in JSON, XML, HTML. Example: <rec><n>Prashant Rao</n><age>35</age></rec>
Unstructured Data
No fixed format or schema. Hardest to analyze with traditional databases. Examples: CCTV footage, social media posts, voice recordings, PDF documents.

Based on Nature

Qualitative (Categorical): Nominal — no order (colors, gender, country); Ordinal — ordered but no fixed interval (Grades: A, B, C; ratings: Good, Better, Best).

Quantitative (Numerical): Discrete — whole numbers (students in a class, cars sold); Continuous — infinite values in a range (height, weight, temperature).

Based on Source & Sensitivity

Primary Data — Collected firsthand for a specific purpose (surveys, experiments). Secondary Data — Already collected and reused (census reports, government databases).

Public — freely available. Private — restricted (employee records). Sensitive — requires legal protection (biometric data, medical records / PII). Confidential — highly restricted (military intelligence, trade secrets).

04 The 5 V's of Big Data

V
Volume
Amount of data generated. Facebook: 500+ TB/day
V
Velocity
Speed of generation. Google: 3.5B searches/day
V
Variety
Diversity of formats. Structured + Semi + Unstructured
V
Veracity
Quality & trustworthiness. Garbage in → Garbage out
V
Value
Meaningfulness. IBM Watson: genomic data → treatment plans

Volume

Refers to the sheer scale of data generated every second from social media, sensors, transactions, and M2M communications. Requires distributed systems like Hadoop — no single machine can store it all. Example: Facebook ingests 500+ terabytes of data daily from photos, messages, and likes.

Velocity

The rate at which data is generated and must be processed. Data flows in continuously — results must appear in milliseconds, not hours. Example: Google handles 3.5 billion+ searches per day — results must appear in milliseconds.

Variety

Data arrives in structured (SQL tables), semi-structured (JSON, XML), and unstructured (images, videos, audio) forms. Traditional RDBMS handles only structured data. Example: A hospital deals with structured patient records, semi-structured lab reports (XML), and unstructured MRI scans — all simultaneously.

Veracity

Big Data can be messy — missing values, duplicates, fake entries, sensor errors. Analyzing unverified data leads to flawed conclusions: garbage in, garbage out. Example: Amazon product reviews contain fake/paid reviews that distort ratings.

Value

Raw data at petabyte scale has zero worth unless transformed into actionable insights. The most important V — all others are meaningless without extracting value. Example: IBM Watson Health analyzed genomic data from cancer patients to recommend personalized treatment plans.

05 Big Data Analytics

Big Data analytics is the process of collecting, organizing, and analyzing large datasets to discover patterns, correlations, and actionable insights. It requires specialized tools for predictive analytics, data mining, text mining, forecasting, and data optimization — collectively called high-performance analytics.

Process of Big Data Analytics

  1. Objective — Define the primary goal. What question are we trying to answer?
  2. Collection — Gather data from diverse sources: databases, social media, sensors, logs, APIs.
  3. Validation — Check data against predefined rules to ensure quality. Since data is massive, it may contain errors, missing values, and inconsistencies.
  4. Preparation — Clean and transform data into a format compatible with the analytical tool.
  5. Analysis — Apply statistical models, ML algorithms, or query engines to extract meaningful information.
  6. Visualization — Convert results into graphs, charts, dashboards, heatmaps.
  7. Communication — Deliver insights to the organization for decision-making.

Importance of Data Validation

  • Accuracy & Reliability — Prevents erroneous conclusions from dirty data.
  • Data Quality — Filters out corrupt, invalid, or irrelevant data at ingestion.
  • Consistency — Ensures data from multiple sources aligns in structure and meaning (date formats, units).
  • Compliance — Ensures data meets HIPAA (healthcare) or GDPR (EU data privacy) regulations.
  • Efficient Decision-Making — Stakeholders can trust the analysis when underlying data is validated.

Challenges & Benefits

Challenges: Breaking data silos across departments; handling unstructured data; scale that defeats traditional methods requiring distributed, parallel processing architectures.

Benefits: A survey of 540 enterprise decision-makers found ~62% of organizations use Big Data analytics to improve speed and reduce operational complexity. France's Orange Telecom anonymized 2.5 billion call records from 5 million users, enabling researchers to track population movement and model disease containment strategies.

07 Data Storage Fundamentals

For Big Data to be stored effectively, the storage model must satisfy these requirements:

  • Scalable — Must expand horizontally (add more machines) as volume grows. You cannot just buy a bigger hard drive — you add more nodes to the cluster.
  • Flexible — Must handle data of any format — structured tables, JSON documents, images, raw text — without a predefined schema.
  • Distributed File System — Data is split and stored across multiple machines, enabling parallel processing.
  • Parallel Processing — Multiple machines process different portions of data simultaneously. Sequential processing of Big Data would be computationally prohibitive.
  • Redundant (Fault Tolerant) — Each data segment is replicated across multiple nodes. Hadoop's default replication factor is 3 — every data block is stored on three separate machines.

08 Hadoop Framework

Hadoop is an open-source distributed computing framework inspired by Google's GFS and MapReduce papers. Designed to store and process massive datasets across clusters of commodity hardware.

HDFS — Hadoop Distributed File System

NameNode (Master) — Stores metadata only. Keeps track of file-to-block mapping, block locations on DataNodes, and DataNode IP addresses. Think of it as a library index — it tells you where a book is, but doesn't contain the book itself.

DataNode (Worker) — Stores actual data in fixed-size Blocks (default 64 MB). A 1 GB file is split into ~16 blocks of 64 MB each, distributed across DataNodes.

Fault Tolerance — Default replication factor of 3. Each block stored on 3 different DataNodes. If one crashes, NameNode detects it via heartbeat signals and instructs re-replication.

Developed by Google internally (2003), GFS inspired Hadoop's HDFS. GFS Master is analogous to HDFS's NameNode. Chunkservers (analogous to DataNodes) store actual data chunks. Clients contact the Master for metadata, then directly communicate with Chunkservers — reducing load on the master. Default chunk size: 64 MB. Designed for large sequential reads, not random access.

Clustering & Classification

01 Introduction to Data Clustering

Clustering is an unsupervised machine learning technique that groups similar data points together without being told the categories in advance — like dumping a box of mixed fruits on a table and naturally grouping apples, bananas, and oranges based on similarity, with no prior labeling. Formally, it partitions a dataset into groups (clusters) such that objects within the same cluster are more similar to each other than to objects in other clusters. Similarity is measured using Euclidean distance, Manhattan distance, or cosine similarity.

Key Properties of Good Clusters

  • High intra-cluster similarity — Points within a cluster should be as close to each other as possible.
  • Low inter-cluster similarity — Points in different clusters should be as far apart as possible.
🌐
Real-World Use Google News uses clustering to group articles about the same event from hundreds of different sources, presenting them as a single story rather than hundreds of redundant links.

02 K-Means Clustering

K-Means is an iterative, partition-based clustering algorithm — like splitting students into study groups: randomly pick 3 "group leaders," every student joins the nearest leader, then each leader moves to the actual average position of their group. Repeat until no one switches. Formally, it minimizes the Within-Cluster Sum of Squares (WCSS), also called inertia:

\[ WCSS = \sum_{i=1}^{k} \sum_{x \in C_i} ||x - \mu_i||^2 \]

where μᵢ is the centroid of cluster Cᵢ

Algorithm Steps

  1. Initialize — Choose k initial centroids randomly (or using smarter K-Means++ initialization).
  2. Assignment Step — Assign every data point to the nearest centroid using Euclidean distance: distance = √[(x₁-x₂)² + (y₁-y₂)²]
  3. Update Step — Recalculate each centroid as the mean of all points currently assigned to it.
  4. Repeat — Repeat steps 2–3 until centroids no longer change (convergence).

Choosing K — The Elbow Method

The Elbow Method plots WCSS against different values of k. As k increases, WCSS decreases. The "elbow" — where the rate of decrease sharply slows — is the optimal k. If the elbow is at k=3, adding a 4th cluster provides diminishing returns.

         ELBOW METHOD — Choosing optimal k
  WCSS
   │
  High ─  *
          │ \
          │  *
          │   *  ← ELBOW (optimal k=3)
          │    ────*────────*────────*──
  Low  ─        k=3   k=4   k=5
         k=1  k=2  k=3  k=4  k=5  k=6

Limitations

  • Must specify k in advance — not always known.
  • Sensitive to outliers — a single extreme point can pull a centroid far from the true cluster center.
  • Assumes clusters are roughly spherical and equally sized — fails for elongated or irregular shapes.
  • Results vary based on initial centroid placement — running K-Means multiple times with different initializations is standard practice.

03 Basics of Classification

Classification is predicting which category a new item belongs to based on labeled examples — like a spam filter trained on thousands of labeled emails that can judge new ones. Formally, it is a supervised machine learning task where an algorithm learns a mapping function f: X → Y from labeled training data, where X is the input feature space and Y is the set of discrete class labels.

Key Terms

Training Set
Labeled data used to train the model.
Test Set
Unseen labeled data used to evaluate model performance.
Feature
An input variable used for prediction. Example: email word count, sender domain.
Label / Class
The output category being predicted. Example: spam / not-spam.
Classifier
The model that performs the classification.

Types of Classification

  • Binary Classification — Two possible classes. Example: Spam detection (spam / not spam), disease diagnosis (positive / negative).
  • Multi-class Classification — More than two classes. Example: Digit recognition (0–9), news categorization.
  • Multi-label Classification — An instance can belong to multiple classes. Example: A movie tagged as both "Action" and "Thriller."

Evaluation Metrics

Accuracy = (TP + TN) / (TP + TN + FP + FN)  |  Precision = TP / (TP + FP)  |  Recall = TP / (TP + FN)

\[ F1 = 2 \times \frac{Precision \times Recall}{Precision + Recall} \]

Harmonic mean of Precision and Recall. A Confusion Matrix summarizes TP, TN, FP, FN.

04 Decision Trees

A Decision Tree works like the game "20 Questions" — start at the top, ask a question, go left or right based on the answer, keep going until you reach a verdict at the bottom. Formally, it's a tree where each internal node represents a test on a feature, each branch represents an outcome, and each leaf node represents a class label.

How Splitting Works — Entropy & Information Gain

\[ Entropy(S) = -\sum_{i=1}^{c} p_i \log_2(p_i) \]

Entropy = 0: perfectly pure. Entropy = 1: maximum uncertainty (evenly split classes).

Information Gain = Entropy(parent) − weighted average Entropy(children). The feature with the highest information gain is selected as the splitting criterion.

Gini Impurity (used in CART): Gini = 1 − Σpᵢ²

Advantages & Disadvantages

  • Advantages: Highly interpretable; handles numerical and categorical features; no feature scaling needed; can model non-linear relationships.
  • Disadvantages: Prone to overfitting — a too-deep tree memorizes training data and fails on new data. Also unstable — small data changes produce very different trees.

Pruning

  • Pre-pruning (Early Stopping) — Stop growing the tree before it perfectly fits training data (e.g., minimum samples per leaf = 5).
  • Post-pruning — Grow the full tree, then trim branches that don't improve performance on validation data.

04b Random Forest

A Random Forest is an ensemble of many Decision Trees — instead of trusting one tree's judgment, you train hundreds of trees on different random subsets of data and features, then let them vote. One biased tree makes mistakes; 500 diverse trees together are far more reliable.

How It Works — Step by Step

  • Step 1 — Bootstrap Sampling (Bagging): From the original dataset of n rows, randomly sample n rows with replacement for each tree. ~37% of rows are left out per tree — called Out-of-Bag (OOB) samples.
  • Step 2 — Feature Randomness: At each node split, only a random subset of m features is evaluated (typically m = √total_features for classification). Forces diversity — different trees focus on different features.
  • Step 3 — Grow Each Tree Fully: Each tree is grown without pruning — deliberately allowed to overfit its bootstrap sample. Individual trees have high variance but low bias.
  • Step 4 — Aggregation (Voting): For a new data point, every tree independently predicts a class. Final prediction = majority vote (classification) or mean (regression).
  • Step 5 — OOB Evaluation: The ~37% of rows not used to train each tree serve as a built-in validation set. OOB error is computed without needing a separate test set.

Why It Beats a Single Decision Tree

  • A single tree overfits — it memorizes noise in training data. Random Forest's diversity averages out individual errors.
  • The key insight: errors of different trees are uncorrelated (because of random feature selection). When errors are uncorrelated, averaging reduces variance dramatically.
  • Mathematically: if each tree has error rate p and errors are independent, ensemble error approaches 0 as number of trees grows.

Feature Importance

  • Random Forest naturally ranks features by how much they improve splits (measured by Gini impurity reduction) averaged across all trees.
  • Useful not just for prediction but for understanding which features matter most — a loan default model might reveal "credit utilization ratio" matters more than "annual income."

05 Naïve Bayes Classifier

Naïve Bayes is a probabilistic classifier based on Bayes' Theorem — like a doctor diagnosing a patient by calculating: "Given these symptoms, which disease is most probable?" The "naïve" part means we assume each feature is completely independent of the others, which is rarely true but works surprisingly well in practice.

\[ P(C|X) = \frac{P(X|C) \cdot P(C)}{P(X)} \]

Posterior = Likelihood × Prior / Evidence. We compare posteriors across classes — P(X) cancels out.

\[ \hat{C} = \arg\max_{C} P(C) \cdot \prod_{i=1}^{n} P(x_i|C) \]

Types of Naïve Bayes

Gaussian NB
Features are continuous, assumed to follow a normal distribution. Used for numerical data like sensor readings.
Multinomial NB
Used for discrete count data. Most common for text classification (word frequencies).
Bernoulli NB
Features are binary (present/absent). For document classification based on word presence, not frequency.

Advantages & Disadvantages

  • Advantages: Extremely fast to train; works well with small training datasets; handles high-dimensional data (thousands of text features); naturally handles multi-class problems.
  • Disadvantages: Independence assumption is almost never true; zero-frequency problem solved by Laplace Smoothing (add 1 to all counts); not a good raw probability estimator.

Association & Recommendation Systems

01 Introduction to Association Rules

Association rule mining finds patterns of the form "if a customer buys X, they also tend to buy Y." The classic example: Walmart discovered in the 1990s that customers who buy diapers frequently also buy beer — no intuitive reason, but the data revealed it. Walmart placed beer near diapers — boosting sales of both. Formally, rules take the form {Antecedent} → {Consequent}, quantified using Support, Confidence, and Lift.

Key Metrics

\[ Support(A \cup B) = \frac{\text{Transactions containing both A and B}}{\text{Total transactions}} \]
\[ Confidence(A \rightarrow B) = \frac{Support(A \cup B)}{Support(A)} \]

Confidence = 0.80 means 80% of the time A is purchased, B is also purchased.

\[ Lift(A \rightarrow B) = \frac{Confidence(A \rightarrow B)}{Support(B)} \]

Lift > 1: positive correlation. Lift = 1: independent. Lift < 1: negative correlation. Lift of 2.5 means A makes B 2.5× more likely.

02 Apriori Algorithm

Apriori is built on one key insight: if an itemset is infrequent, any larger set containing it will also be infrequent — so we prune it early. We build upward: start with single items, keep only frequent ones, generate pairs, keep frequent pairs, then triplets — like a pyramid where only items that pass each layer advance. This is the Apriori property (downward closure / anti-monotone property).

💡
Apriori Property If an itemset is infrequent, ALL its supersets are also infrequent. This lets us prune the search space dramatically.

Algorithm Steps

  1. Generate L₁ — Scan database, count each item. Keep items with support ≥ min_sup.
  2. Generate C₂ — Join frequent 1-itemsets with themselves to generate candidate pairs.
  3. Prune C₂ — Remove candidates whose subsets are not in L₁.
  4. Generate L₂ — Scan database, count C₂ candidates. Keep those with support ≥ min_sup.
  5. Repeat — Until no more frequent itemsets can be found.
  6. Generate Rules — For each frequent itemset, generate all possible rules. Keep rules with confidence ≥ min_conf.

FP FP-Growth (Frequent Pattern Growth)

FP-Growth eliminates Apriori's two biggest bottlenecks: multiple database scans and massive candidate generation. The core idea: compress the entire transaction database into a compact tree structure called an FP-Tree in just 2 database scans, then mine frequent patterns directly from the tree — never generating candidate itemsets.

Why FP-Growth over Apriori?

  • Apriori scans the database once per itemset size level (k scans for k-length itemsets).
  • FP-Growth always uses exactly 2 database scans regardless of pattern length.
  • No candidate generation means no memory wastage on infrequent itemsets.
  • In practice, FP-Growth is 10–100x faster than Apriori on dense datasets.

Key Concepts

FP-Tree
A compressed, prefix-tree representation of the transaction database. Shared prefixes are merged, making the tree much smaller than the original database.
Header Table
Lists all frequent items sorted by frequency descending. Each entry links all nodes of the same item together via node-link pointers.
Conditional Pattern Base
For item X, all prefix paths in the FP-Tree ending at a node labeled X. Tells you "what items always appear alongside X."
Conditional FP-Tree
An FP-Tree built from X's Conditional Pattern Base. Mining this tree gives all frequent itemsets containing X.

Algorithm Steps

  1. Scan 1: Count support of every item. Remove below min_sup. Sort remaining by frequency descending — this is the frequent item order.
  2. Build FP-Tree (Scan 2): Re-read each transaction, keep only frequent items, sort in frequent item order, insert into tree. Shared prefixes are merged (count incremented). Build Header Table simultaneously.
  3. Mine the FP-Tree: For each item (least frequent first), find Conditional Pattern Base, build Conditional FP-Tree, extract frequent itemsets recursively.

04 MapReduce for Association Analysis

Association rule mining on billions of transactions (Walmart, Amazon) cannot run on a single machine — the dataset can't even fit in memory. MapReduce distributes computation across a cluster.

MapReduce Paradigm

Map Phase — Each mapper receives a partition of the data and emits (key, value) pairs — e.g., each item in a transaction as key with value 1.

Shuffle & Sort Phase — Framework automatically groups all values by key, so all counts for the same item arrive at the same reducer.

Reduce Phase — Each reducer receives a key and a list of values, and aggregates them — summing all 1s to get total support count.

Round 1 (L₁): Map: emit (item, 1) for each item. Reduce: sum counts, output items ≥ min_sup.

SON Algorithm (Savasere, Omiecinski, Navathe) — Optimized two-pass algorithm: Pass 1 finds frequent itemsets locally in each chunk (using min_sup scaled to chunk size). Pass 2 counts global support across the full dataset using local frequent itemsets as candidates.

05 Recommendation Systems

Types

Content-Based Filtering
Recommends items similar to items the user previously liked, based on item features. Pandora's Music Genome Project analyzes 450+ musical attributes per song and recommends based on attributes of songs you've liked.

Challenges

  • Scalability — Computing similarity between all user pairs is O(n²). Solution: Approximate nearest neighbor algorithms like Locality Sensitive Hashing (LSH) or distributed Spark.

Stream Data & Real-Time Analytics

01 Understanding Stream Data

Traditional data processing is like filling a bucket, then analyzing it when full. Stream data processing is like measuring the flow as it comes out of the tap — you analyze it in real time, as it flows, without waiting for it to accumulate. Formally, data streams are sequences of records generated continuously and rapidly from IoT sensors, social media feeds, financial markets, clickstreams, and log files.

Key Characteristics

  • Unbounded — There is no defined end to the stream.
  • Ordered — Records arrive in time order (or approximately so).
  • Non-revisitable — In pure streaming systems, you cannot go back and reprocess old data.
  • High velocity — Data arrives faster than it can be stored persistently in traditional databases.

Sliding Windows

Since stream data is infinite, analysis is performed over windows — finite subsets of the stream:

  • Tumbling Window — Fixed-size, non-overlapping time windows. Example: Aggregate Twitter mentions every 5 minutes; the window resets after each period.
  • Sliding Window — Overlapping windows that move at a fixed step. Example: Count tweets in the last 10 minutes, updated every 1 minute.
  • Session Window — Opens when user activity begins and closes after a period of inactivity.

02 Introduction to Stream Computing

The fundamental limitation of batch processing (Hadoop MapReduce) is latency. For fraud detection, you cannot detect a fraudulent credit card transaction 30 minutes after it happened. Stream computing processes data as it arrives, enabling responses in milliseconds to seconds.

Key Stream Processing Frameworks

Apache Kafka
Distributed, fault-tolerant message streaming platform. Producers publish to topics; consumers subscribe and read in real time. Kafka persists messages on disk for configurable periods (replay capability). LinkedIn originally developed Kafka to handle 7 trillion messages/day. Architecture: Producer → Topic (Partitions) → Broker → Consumer → Zookeeper.
Apache Spark Streaming
Extension of Spark. Uses a micro-batch model: incoming data is divided into small batches (e.g., every 1 second) and processed using Spark's batch engine. Structured Streaming (newer API) treats a stream as an unbounded table that continuously grows.
Apache Flink
True event-driven stream processing — processes each event individually as it arrives (unlike Spark's micro-batch). Achieves genuine real-time with low latency. Alibaba uses Flink to process 4.4 trillion events/day during Singles' Day.
Apache Storm
One of the earliest real-time frameworks. Uses a topology model: a DAG of Spouts (data sources) and Bolts (processing units). Twitter developed Storm to process trending topics in real time.

Lambda Architecture

A common Big Data architecture combining batch and stream processing:

  • Batch Layer — Recomputes results on full historical dataset periodically. Slow but accurate. Hadoop/Spark batch jobs.
  • Speed Layer — Processes real-time stream data for low-latency results. Approximate but fresh. Apache Storm/Flink.
  • Serving Layer — Merges results from both layers to answer queries. Apache Cassandra, HBase.

04 Introduction to Graph Analytics

A graph is a network of connected entities — Facebook is a graph of people (nodes) connected by friendships (edges); the internet is a graph of web pages connected by hyperlinks. Formally, a graph G = (V, E) consists of Vertices (V) — entities — and Edges (E) — relationships. Graphs can be Directed (A → B ≠ B → A, like Twitter followers), Undirected (A—B, like Facebook friendships), or Weighted (edges with numerical values like road distances).

Key Algorithms

\[ PR(A) = \frac{1-d}{N} + d \sum_{i \in Links(A)} \frac{PR(i)}{L(i)} \]

PageRank — d is damping factor (0.85), N is total pages, L(i) is outbound links from page i. Foundation of Google Search.

  • Shortest Path (Dijkstra's Algorithm) — Finds the minimum-cost path between two nodes. Google Maps uses variants of Dijkstra's and A* to find fastest routes in real time, factoring in live traffic.
  • Community Detection (Louvain Algorithm) — Groups densely interconnected nodes. Facebook uses community detection to identify social circles; posts from within your community rank higher in News Feed.
  • Degree Centrality — Number of edges connected to a node (friends count).
  • Betweenness Centrality — How often a node lies on the shortest path between other nodes. Used in epidemiology to identify super-spreaders.
  • Eigenvector Centrality — A node is important if connected to other important nodes. PageRank is a variant.

NoSQL Data Management

01 NoSQL Databases — Introduction

Relational databases are like a perfectly organized spreadsheet — every row has the same columns, and everything fits neatly. This is great for bank records but terrible for storing a tweet (0 to 100 hashtags), a product catalog (each product has entirely different attributes), or a social network (where relationships matter more than the data itself). NoSQL (Not Only SQL) refers to a broad class of database systems that abandon fixed tabular schemas in favor of flexibility, horizontal scaling, and distributed architecture.

Why NoSQL?

  • Fixed Schema — Adding a new attribute to a table with billions of rows in RDBMS requires an expensive ALTER TABLE operation that can take hours and lock the table.
  • Vertical Scaling Only — SQL databases scale by upgrading hardware. NoSQL scales horizontally — add more commodity machines to the cluster.
  • Joins are Expensive at Scale — JOINing tables with billions of rows is computationally intensive. NoSQL avoids joins by denormalizing data.
  • Unstructured Data — RDBMS cannot natively store JSON, arrays, or nested objects. NoSQL handles these naturally.

CAP CAP Theorem

The CAP Theorem (Brewer's Theorem) states that a distributed system can guarantee at most 2 of 3 properties simultaneously:

  • Consistency (C) — Every read receives the most recent write (or an error). All nodes see the same data at the same time.
  • Availability (A) — Every request receives a response (not necessarily the most recent data).
  • Partition Tolerance (P) — The system continues to operate even if some nodes cannot communicate (network partition).

Since network partitions are unavoidable in distributed systems, systems must choose between CP (consistent but may be unavailable during partition) and AP (always available but may return stale data).

02 Schema-less Models and Their Flexibility

In relational databases, schemas are rigid and predefined. In NoSQL databases, schemas are dynamic — each record can have different fields. You do not need to define the structure in advance.

  • Rapid iteration — Developers can add new fields to documents without database migrations.
  • Heterogeneous data — Different records in the same collection can have completely different shapes. An e-commerce database might have electronics with voltage/wattage fields and clothing with size/color fields — in the same collection.
  • JSON-native storage — Most NoSQL document stores natively store data as JSON or BSON, matching web API formats.
  • Tradeoff — Schema-less databases push validation responsibility to the application layer. No database-enforced constraints.

Mongoose (ODM for MongoDB in Node.js) adds optional schema enforcement back to MongoDB — giving NoSQL flexibility with some SQL-like structure guarantees.

03 Key-Value Stores

A key-value store is like a giant dictionary or hashmap — you store data under a unique key, and to retrieve it, you just provide that key. The database doesn't know or care what the value contains; it stores and retrieves it at extremely high speed. Operations: PUT(key, value), GET(key) → value, DELETE(key).

When to Use

  • Data access is always by primary key — you always know the exact key.
  • You need extremely fast reads and writes — no parsing, no query planning.
  • The data is self-contained per key — no need to query across keys.

Examples

Redis
In-memory key-value store with support for complex value types: strings, lists, sets, sorted sets, hashes. Microsecond latency (stored in RAM). Used as a caching layer in front of slower databases. Twitter caches timelines in Redis — each user's timeline is a sorted set, delivering sub-millisecond tweet feeds.
Amazon DynamoDB
Fully managed, distributed key-value store. Guarantees single-digit millisecond latency at any scale. Samsung uses DynamoDB to handle 4M+ requests/second from IoT devices during peak hours.
Apache Cassandra
(Also a wide-column store) Originally designed at Facebook for the Inbox search feature. Optimized for write-heavy workloads. Netflix uses Cassandra to store viewing history and manage session data for 200M+ subscribers globally.

Limitations

  • No complex queries — cannot filter by value or perform JOINs.
  • No relationships between keys — data model must be designed around key access patterns.
  • Requires denormalization — the same data may need to be stored under multiple keys for different access patterns.

04 Document Stores

Document stores are key-value stores where the "value" is a structured document (usually JSON) that the database understands and can query inside. Instead of a flat key → opaque blob, you store rich nested objects and can query any field within them — like a self-describing record that contains everything you need without JOINing other tables.

MongoDB

The most widely used document database. Database → Collections → Documents (BSON objects). No fixed schema — documents in the same collection can have completely different fields, but any field can be indexed.

05 Tabular (Wide-Column) Data Stores

Wide-column stores are like a spreadsheet where each row can have completely different columns — and there can be millions of them. Each row is identified by a row key, and under it, you store any set of column-value pairs. Ideal for time-series data (each timestamp is a column) and sparse data. Formally:

Row Key → Column Family → Column Qualifier → Value (with timestamp)

Each cell is timestamped, allowing multiple versions to coexist. Column families are predefined at schema creation, but columns within a family are added dynamically at write time.

Apache HBase

Open-source wide-column store built on HDFS, modeled after Google BigTable. HMaster manages region assignment (analogous to HDFS NameNode). RegionServers manage regions (contiguous row key ranges). ZooKeeper coordinates distributed components.

💬
Real-World Use Facebook originally used HBase to power its Messages feature, storing billions of messages. At peak, HBase handled over 20 petabytes of message data.

Google BigTable

Google's proprietary wide-column store (2006 paper by Chang et al.). Powers Google Analytics (website traffic for billions of sites), Google Earth (satellite imagery tiles keyed by coordinates and zoom level), and Gmail. Key insight: using row key, column family, qualifier, and timestamp as a multi-dimensional key achieves efficient range scans.

Apache Cassandra

Distributed, leaderless wide-column store developed at Facebook (2008). Designed for high write throughput and high availability with no single point of failure.

  • Peer-to-peer architecture — No master node. Every node is equal. Data replicated automatically.
  • Tunable Consistency — Choose consistency level per query: ONE (fastest) to ALL (slowest). QUORUM is common middle ground.
  • Ring topology — Nodes arranged in a logical ring. Data distributed using consistent hashing on the partition key.
  • CQL (Cassandra Query Language) — SQL-like syntax but with NoSQL semantics. No JOINs or subqueries.

06 Graph Databases

Graph databases store data as a network of nodes and edges — finding "friends of friends" in a relational database requires expensive multi-level JOINs; in a graph database, you simply traverse the edges, like following paths on a map instead of doing address lookups in a spreadsheet. Native graph processing systems like Neo4j store relationships directly on disk adjacent to the nodes they connect, enabling index-free adjacency: to find connected nodes, you follow a pointer — no index lookup required.

Neo4j and Cypher Query Language

// Find all friends of Alice who also know Bob
MATCH (alice:Person {name: "Alice"})-[:FRIEND]->(friend)-[:FRIEND]->(bob:Person {name: "Bob"})
RETURN friend.name

// Find the shortest path between Alice and Charlie
MATCH path = shortestPath((alice:Person {name: "Alice"})-[*]-(charlie:Person {name: "Charlie"}))
RETURN path
🗃️
Real-World Use Panama Papers investigation — Journalists at ICIJ used Neo4j to analyze 11.5 million leaked documents about offshore tax havens. The network of companies, people, and accounts — impossible to analyze with spreadsheets — was mapped as a graph, revealing links between world leaders and offshore accounts.

Fraud Detection
A single fraudulent transaction looks innocent in isolation, but a graph reveals the same phone number, address, or device connected to dozens of different accounts. PayPal uses graph-based fraud detection to protect hundreds of millions of transactions.
